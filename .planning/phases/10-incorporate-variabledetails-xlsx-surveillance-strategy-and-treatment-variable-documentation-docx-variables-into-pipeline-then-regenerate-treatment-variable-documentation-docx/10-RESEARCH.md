## Summary

This phase extends the R pipeline in four areas: (1) surveillance modality detection via CPT/HCPCS/LOINC codes from VariableDetails.xlsx into new 13_surveillance.R, (2) survivorship encounter classification by joining PROVIDER table specialty codes into new 14_survivorship_encounters.R, (3) timing derivation variables (DAYS_DX_TO_*) added inline to 04_build_cohort.R, and (4) auto-generated comprehensive documentation via new 15_generate_documentation.R that reads pipeline code and emits both markdown and docx.

The existing pipeline provides all the patterns needed: detection functions return `tibble(ID, HAD_*)`, code lists live in 00_config.R named lists, and 11_generate_pptx.R already uses the `officer` package for document generation. The main new technical requirements are loading two new PCORnet CDM tables (LAB_RESULT_CM with LOINC-based lab matching, PROVIDER with NUCC taxonomy specialty matching), both of which follow established col_types patterns.

The docx generation decision favors the two-step approach: write a markdown `.md` file programmatically (using plain R string operations and cat/writeLines), then convert to docx using `rmarkdown::render()` with `output_format = word_document()`. This avoids officer's imperative API for purely text-based documentation content and keeps the `.md` as the durable source of truth.

**Primary recommendation:** Add new scripts 13_surveillance.R, 14_survivorship_encounters.R, 15_generate_documentation.R; extend 00_config.R with SURVEILLANCE_CODES/LAB_CODES/SURVIVORSHIP_CODES/PROVIDER_SPECIALTIES lists; add LAB_RESULT_CM and PROVIDER to 01_load_pcornet.R; integrate flags into 04_build_cohort.R.

### Pattern 1: Surveillance Detection Function (mirrors has_chemo pattern)
**What:** Function returning tibble(ID, HAD_{MODALITY}, FIRST_{MODALITY}_DATE, N_{MODALITY}) for a single modality.
**When to use:** One function per modality (9 modalities + ~7 lab types). Called from assemble_surveillance_flags().
**Example:**
```r
# Source: 03_cohort_predicates.R pattern (has_chemo / has_radiation)
detect_mammogram <- function(post_dx_date_map) {
  # post_dx_date_map: tibble(ID, first_hl_dx_date)
  px_hits <- pcornet$PROCEDURES %>%
    filter(
      (PX_TYPE == "CH" & PX %in% SURVEILLANCE_CODES$mammogram_cpt) |
      (PX_TYPE == "10" & PX %in% SURVEILLANCE_CODES$mammogram_icd10pcs)
    ) %>%
    inner_join(post_dx_date_map, by = "ID") %>%
    filter(PX_DATE > first_hl_dx_date) %>%   # D-03: post-diagnosis only
    group_by(ID) %>%
    summarise(
      FIRST_MAMMOGRAM_DATE = min(PX_DATE, na.rm = TRUE),
      N_MAMMOGRAM = n(),
      .groups = "drop"
    )

  # Build full result with HAD_ flag (0/1 via coalesce on left_join)
  post_dx_date_map %>%
    select(ID) %>%
    left_join(px_hits, by = "ID") %>%
    mutate(
      HAD_MAMMOGRAM = as.integer(!is.na(FIRST_MAMMOGRAM_DATE)),
      N_MAMMOGRAM = coalesce(N_MAMMOGRAM, 0L)
    )
}
```

### Pattern 2: Lab Detection via LAB_RESULT_CM LOINC Matching
**What:** Filter LAB_RESULT_CM by LAB_LOINC codes, restrict post-diagnosis, summarise per patient.
**When to use:** All lab-type surveillance (CRP, liver function, platelets, FOBT — D-02).
**Example:**
```r
# LAB_RESULT_CM key column: LAB_LOINC (uppercase per PCORnet CDM convention in this project)
# Also check: LAB_PX (CPT for ordered lab), RESULT_NUM, RESULT_UNIT
detect_crp <- function(post_dx_date_map) {
  if (is.null(pcornet$LAB_RESULT_CM)) {
    message("[Surveillance] LAB_RESULT_CM not loaded, skipping CRP")
    return(post_dx_date_map %>% select(ID) %>%
           mutate(HAD_CRP = 0L, FIRST_CRP_DATE = as.Date(NA), N_CRP = 0L))
  }
  lab_hits <- pcornet$LAB_RESULT_CM %>%
    filter(LAB_LOINC %in% LAB_CODES$crp_loinc) %>%
    inner_join(post_dx_date_map, by = "ID") %>%
    filter(SPECIMEN_DATE > first_hl_dx_date | LAB_ORDER_DATE > first_hl_dx_date) %>%
    group_by(ID) %>%
    summarise(
      FIRST_CRP_DATE = min(coalesce(SPECIMEN_DATE, LAB_ORDER_DATE), na.rm = TRUE),
      N_CRP = n(),
      .groups = "drop"
    )
  post_dx_date_map %>% select(ID) %>%
    left_join(lab_hits, by = "ID") %>%
    mutate(HAD_CRP = as.integer(!is.na(FIRST_CRP_DATE)), N_CRP = coalesce(N_CRP, 0L))
}
```

### Pattern 3: Survivorship Level Classification via PROVIDER Join
**What:** Join ENCOUNTER to PROVIDER via PROVIDERID, check PROVIDER_SPECIALTY_PRIM against cancer provider taxonomy codes; layer levels 1-4 as per-patient summary.
**When to use:** 14_survivorship_encounters.R — all 4 encounter levels.
**Example:**
```r
# Level 3 adds provider specialty filter on top of level 2
classify_cancer_provider_visits <- function(cancer_related_encounters) {
  # cancer_related_encounters: result of level 2 (AV+TH with HL DX on encounter)
  if (is.null(pcornet$PROVIDER)) {
    message("[Survivorship] PROVIDER table not loaded, skipping levels 3/4")
    return(cancer_related_encounters %>% mutate(ENC_CANCER_PROVIDER = 0L))
  }
  cancer_related_encounters %>%
    left_join(
      pcornet$PROVIDER %>% select(PROVIDERID, PROVIDER_SPECIALTY_PRIM),
      by = "PROVIDERID"
    ) %>%
    filter(PROVIDER_SPECIALTY_PRIM %in% PROVIDER_SPECIALTIES$cancer_oncology)
}
```

### Anti-Patterns to Avoid
- **Post-treatment date filtering instead of post-diagnosis:** D-03 is explicit: surveillance detection uses first_hl_dx_date as anchor, not first treatment date. Do not reuse compute_payer_mode_in_window() logic here.
- **Loading LAB_RESULT_CM eagerly in all scripts:** LAB_RESULT_CM is large. Load only via 01_load_pcornet.R, access via `pcornet$LAB_RESULT_CM` with null-safe guard.
- **Hard-coding NUCC specialty strings:** Store codes and display names in PROVIDER_SPECIALTIES list in 00_config.R, reference by name in functions. Makes it auditable.
- **Building officer document node-by-node for text-heavy content:** For the variable documentation (pure text/tables), use rmarkdown::render() to avoid officer's imperative body_add_* for every paragraph.
- **Treating LAB_LOINC_SOURCE vs LAB_LOINC confusion:** The key matching column is LAB_LOINC (the LOINC code itself). LAB_LOINC_SOURCE describes where the LOINC code came from (in-house vs registry). Match only on LAB_LOINC.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown to docx conversion | Custom string builder for XML Word format | `rmarkdown::render(..., word_document())` | Pandoc handles all Office XML complexity; reference_docx templates for styling |
| NUCC taxonomy code lookup | Parse NUCC PDF at runtime | Store verified codes in PROVIDER_SPECIALTIES list in 00_config.R | Codes rarely change; runtime PDF parsing is fragile and slow |
| Date difference calculation | Custom interval function | `as.integer(date2 - date1)` via lubridate/base R Date subtraction | lubridate Date objects subtract cleanly to difftime, cast to integer gives days |
| LOINC version lookup | API call to Regenstrief LOINC service | Use LOINC codes from VariableDetails.xlsx "Labs" sheet directly | Researcher has verified codes in spreadsheet; no runtime lookup needed |

## Common Pitfalls

### Pitfall 1: post_dx_date_map Not Filtered to Cohort Patients
**What goes wrong:** If detect_* functions operate on all rows in PROCEDURES/LAB_RESULT_CM without restricting to cohort IDs first, results include non-cohort patients — inflating N and potentially joining incorrectly to hl_cohort.
**Why it happens:** LAB_RESULT_CM likely contains records for all patients in the extract, not just HL cohort members.
**How to avoid:** Pass `post_dx_date_map` (tibble of cohort ID + first_hl_dx_date) as function argument, use `inner_join(post_dx_date_map, by = "ID")` before date filtering. This simultaneously restricts to cohort and provides the anchor date.
**Warning signs:** HAD_{MODALITY} count exceeds total hl_cohort rows.

### Pitfall 2: PROVIDERID Missing in ENCOUNTER for Many Rows
**What goes wrong:** Level 3/4 encounter classification requires PROVIDERID in ENCOUNTER to join to PROVIDER table, but PROVIDERID can be NULL for outpatient encounters or ambulatory records in PCORnet.
**Why it happens:** PROVIDERID is not required in ENCOUNTER per CDM spec; it's optional.
**How to avoid:** Use `left_join` (not `inner_join`) when joining ENCOUNTER to PROVIDER. Rows with NULL PROVIDERID will not match cancer provider specialties, which is correct — they stay at level 2, not level 3. Log count of encounters with NULL PROVIDERID.
**Warning signs:** ENC_CANCER_PROVIDER count = 0 when ENC_CANCER_RELATED > 0 (suggests PROVIDERID missing entirely from extract).

### Pitfall 3: LAB_LOINC Null/Missing Despite Lab Record Existing
**What goes wrong:** Sites may populate RAW_LAB_CODE but leave LAB_LOINC as NA when local LOINC mapping is incomplete. Filtering only on LAB_LOINC misses these records.
**Why it happens:** PCORnet CDM allows LAB_LOINC to be null when only a local code is available; sites should map but not all do.
**How to avoid:** Log `sum(is.na(pcornet$LAB_RESULT_CM$LAB_LOINC))` after load. For this phase, LOINC-only matching is correct per D-02 (no local code fallback specified). Document coverage limitation in generated documentation.
**Warning signs:** All HAD_{LAB_*} flags = 0 despite clinical expectation of lab results.

### Pitfall 4: Survivorship ICD Level 4 Codes Span ICD-9 and ICD-10
**What goes wrong:** Personal history codes split across coding eras: V87.41-V87.46/V15.3 are ICD-9 and Z92.21-Z92.25/Z92.3 are ICD-10. Filtering without DX_TYPE will miss the other era.
**Why it happens:** HL cohort spans pre-2015 (ICD-9) and post-2015 (ICD-10) encounter dates.
**How to avoid:** Use DX_TYPE-conditioned filter exactly as TREATMENT_CODES patterns do:
```r
filter(
  (DX_TYPE == "09" & DX %in% SURVIVORSHIP_CODES$personal_history_icd9) |
  (DX_TYPE == "10" & DX %in% SURVIVORSHIP_CODES$personal_history_icd10)
)
```
**Warning signs:** ENC_SURVIVORSHIP = 0 for all patients; or count dramatically lower than ENC_CANCER_PROVIDER.

### Pitfall 5: LAB_RESULT_CM Memory Footprint
**What goes wrong:** LAB_RESULT_CM is typically the largest PCORnet CDM table (millions of rows). Loading it with the same full-load strategy as PROCEDURES may exhaust memory.
**Why it happens:** Routine labs for all patients accumulate quickly.
**How to avoid:** Load via existing `load_pcornet_table()` with null-safe guard. vroom's lazy loading mitigates this. If memory pressure occurs, the null-safe guard (`if (!is.null(pcornet$LAB_RESULT_CM))`) lets the researcher simply omit loading it. The pipeline must be resilient to a missing LAB_RESULT_CM (all lab HAD_* flags = 0).
**Warning signs:** R session crashes during 01_load_pcornet.R after LAB_RESULT_CM load.

### Pitfall 6: Documentation Script Dependencies
**What goes wrong:** 15_generate_documentation.R calls `source("R/00_config.R")` and reads config lists, but if surveillance code lists are defined in a separate file, they won't be available.
**Why it happens:** Split config creates implicit ordering dependencies.
**How to avoid:** Keep all code lists (SURVEILLANCE_CODES, LAB_CODES, SURVIVORSHIP_CODES, PROVIDER_SPECIALTIES) in 00_config.R. Documentation script then only needs `source("R/00_config.R")` to access everything.
**Warning signs:** Error "object 'SURVEILLANCE_CODES' not found" in 15_generate_documentation.R.

# Source: established pattern in 03_cohort_predicates.R
detect_echo <- function(post_dx_date_map) {
  results <- post_dx_date_map %>%
    select(ID) %>%
    mutate(HAD_ECHO = 0L, FIRST_ECHO_DATE = as.Date(NA), N_ECHO = 0L)

  if (!is.null(pcornet$PROCEDURES)) {
    px_hits <- pcornet$PROCEDURES %>%
      filter(PX_TYPE == "CH" & PX %in% SURVEILLANCE_CODES$echo_cpt) %>%
      inner_join(post_dx_date_map, by = "ID") %>%
      filter(PX_DATE > first_hl_dx_date) %>%
      group_by(ID) %>%
      summarise(FIRST_ECHO_DATE = min(PX_DATE, na.rm = TRUE), N_ECHO = n(), .groups = "drop")

    results <- post_dx_date_map %>%
      select(ID) %>%
      left_join(px_hits, by = "ID") %>%
      mutate(HAD_ECHO = as.integer(!is.na(FIRST_ECHO_DATE)), N_ECHO = coalesce(N_ECHO, 0L))
  }

  n_had <- sum(results$HAD_ECHO)
  message(glue("[Surveillance] detect_echo: {n_had} patients with echocardiogram post-diagnosis"))
  results
}
