## Summary

Phase 11 is a targeted modification of a single existing file (`R/11_generate_pptx.R`) with a companion new-slide integration from an already-complete file (`R/16_encounter_analysis.R`). The code base is fully in hand — no new libraries are needed, and all supporting data columns (`AGE_GROUP`, `DX_YEAR`, `HAS_POST_TX_ENCOUNTERS`, `N_ENC_NONACUTE_CARE`, `N_ENCOUNTERS`) are already computed in `hl_cohort` by `04_build_cohort.R`.

The two primary changes are: (1) collapsing 9 raw payer categories to 6 display categories by mapping "Other", "Unavailable", and "Unknown" to "Missing" in `rename_payer()` and updating `PAYER_ORDER`, and (2) adding encounter analysis slides drawn from the figures and summary tables already produced by `16_encounter_analysis.R`. Column totals (PPTX-02) and age groups (PPTX-04) are described as already implemented in CONTEXT.md — research confirms the underlying data infrastructure is present but the PPTX slide for age-group breakdown does not yet exist in `11_generate_pptx.R`.

All work stays in R, using `officer` + `flextable` for PPTX construction and `ggplot2` for figures. The encounter analysis figures are saved as PNG files by `16_encounter_analysis.R` — the decision for Phase 11 (left to Claude's discretion per CONTEXT.md) is whether to embed those PNGs directly into new PPTX slides or re-derive the summary data inline and render with flextable. Research below recommends embedding PNGs as the lower-risk approach.

**Primary recommendation:** Edit `11_generate_pptx.R` in place — update `rename_payer()` + `PAYER_ORDER`, add 4 new encounter-analysis slides (Slides 17-20), audit every existing slide title/subtitle for clarity, and confirm `build_enr_coverage_table` and related functions also iterate over the new 6-category order.

### Pattern 1: Payer Label Consolidation (Display-Only)

**What:** `rename_payer()` maps raw category strings to display labels. Expand to map three categories to "Missing".

**When to use:** Applied once via `mutate(across(..., rename_payer))` — all downstream table builders automatically get the 6-category view.

**Current code (lines 61-67):**
```r
rename_payer <- function(x) {

    x == "No payment / Self-pay" ~ "Self-pay",
    is.na(x) ~ "Unknown",
    TRUE ~ x
  )
}
```

**Required change:**
```r
rename_payer <- function(x) {
  case_when(
    x == "No payment / Self-pay"  ~ "No payment / Self-pay",  # keep full name or shorten

}
```

### Pattern 2: Embedding PNG Figures in PPTX Slides

**What:** `officer::ph_with(external_img(...), location = ph_location(...))` places a raster image in a slide placeholder.

**When to use:** For encounter analysis slides where the figure was already rendered to PNG by 16_encounter_analysis.R.

**Example:**
```r
add_image_slide <- function(pptx, title, subtitle, img_path, img_width = 8, img_height = 5) {
  pptx <- pptx %>%
    add_slide(layout = "Blank") %>%
    ph_with(
      value = fpar(ftext(title, prop = fp_text(font.size = 22, bold = TRUE,
                                               font.family = "Calibri", color = UF_BLUE))),
      location = ph_location(left = 0.5, top = 0.2, width = 9, height = 0.5)
    ) %>%
    ph_with(
      value = fpar(ftext(subtitle, prop = fp_text(font.size = 12, italic = TRUE,
                                                  font.family = "Calibri", color = DARK_TEXT))),
      location = ph_location(left = 0.5, top = 0.65, width = 9, height = 0.3)
    ) %>%
    ph_with(
      value = external_img(img_path, width = img_width, height = img_height),
      location = ph_location(left = 0.5, top = 1.1, width = img_width, height = img_height)
    )
  pptx
}
```

**Source:** officer package documentation — `external_img()` for raster images, `ph_location()` for absolute positioning.

## Common Pitfalls

### Pitfall 1: Slide 15 "Unknown" Filter Breaks After Rename

**What goes wrong:** Line 958 filters `POST_TREATMENT_PAYER == "Unknown"`. After `rename_payer()` maps Unknown to "Missing", this filter returns 0 rows, producing an empty Slide 15.

**Why it happens:** The slide was built before the rename_payer consolidation was designed.

**How to avoid:** Update line 958 to `POST_TREATMENT_PAYER == "Missing"` and update the slide title accordingly.

**Warning signs:** Slide 15 shows only a Total row with 0 patients.

### Pitfall 2: POST_TREATMENT_PAYER Asymmetric Rename Logic

**What goes wrong:** Lines 386-390 apply full `rename_payer()` to primary payer columns, but apply only the Self-pay rename to POST_TREATMENT_PAYER columns (so NA remains NA, enabling "N/A (No Follow-up)" to work correctly). If this logic is collapsed into a single `rename_payer()` call, the N/A (No Follow-up) row disappears.

**Why it happens:** NA post-treatment payer means "no follow-up data", which is meaningfully different from a "Missing" insurance category.

**How to avoid:** Keep the two-step mutation pattern. The POST_TREATMENT_PAYER block must map `Other/Unavailable/Unknown` to "Missing" while leaving NA as NA:
```r
across(
  c(POST_TREATMENT_PAYER, POST_CHEMO_PAYER, POST_RAD_PAYER, POST_SCT_PAYER),
  ~ case_when(
      .x == "No payment / Self-pay"          ~ "No payment / Self-pay",
      .x %in% c("Other", "Unavailable", "Unknown") ~ "Missing",
      TRUE ~ .x  # preserves NA as NA
    )
)
```

**Warning signs:** "N/A (No Follow-up)" row shows 0 count on Slides 3, 5, 7, 9 when you know some patients had no follow-up.

### Pitfall 3: PNG File Path Dependency

**What goes wrong:** `11_generate_pptx.R` is run without first running `16_encounter_analysis.R`, leaving PNG files absent. `external_img()` will throw an error if the file does not exist.

**Why it happens:** The scripts are independent; no automatic dependency chain forces order.

**How to avoid:** Wrap each image slide in `if (file.exists(img_path))` with a clear message directing the user to run `16_encounter_analysis.R` first.

**Warning signs:** R error like "file does not exist" or "cannot open connection".

### Pitfall 4: PAYER_ORDER Length Mismatch in Summary Count Messages

**What goes wrong:** Console messages like "Medicare: 45, Medicaid: 23 ..." that enumerate all 9 categories will now skip rows for Other/Unavailable/Unknown since those values are remapped before the count. Net patient count may appear lower.

**Why it happens:** If any counting code iterates over the old PAYER_ORDER or uses `count(PAYER_CATEGORY_PRIMARY)` on the pre-renamed column.

**How to avoid:** All production counting happens after the `mutate(across(..., rename_payer))` step in `cohort_full`. The rename happens in Section 2e. Any diagnostic `message()` calls that count payer distribution will correctly show the 6-category breakdown automatically.

### Pitfall 5: "No Information" Mapping

**What goes wrong:** CONTEXT.md mentions "No Information" as a category to collapse. The raw `PAYER_MAPPING$categories` in `00_config.R` contains no "No Information" entry. Attempting to add it to `rename_payer()` has no effect.

**Why it happens:** "No Information" is the PCORnet raw code "NI" — it is a `sentinel_value` and triggers fallback to secondary payer during harmonization in `02_harmonize_payer.R`. If both primary and secondary are NI, the effective payer could be NA or map to "Unavailable". It will never appear as the literal string "No Information" in `hl_cohort`.

**How to avoid:** The NA branch in `rename_payer()` handles the NI→NA→"Missing" path. No additional mapping needed.
