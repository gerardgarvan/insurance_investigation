## Summary

Phase 14 involves two distinct activities: (1) conversational review of value_audit CSVs to identify coding inconsistencies suggesting data capture problems, and (2) style-preserving code optimization across the full R pipeline (scripts 01-17) to eliminate redundant operations while maintaining the named predicate + dplyr chain pattern.

The value audit review is a manual, conversational process where Claude reads the CSVs generated by `R/17_value_audit.R` (run by user on HiPerGator) and discusses potential coding inconsistencies — same concepts coded differently across tables/columns (e.g., 'M'/'Male'/'MALE', mixed date formats, ICD code format mismatches). This is exploratory data quality assessment, not automated validation.

Code optimization focuses on reducing unnecessary data copies, simplifying redundant logic, improving column selection, and optimizing vroom usage — all while preserving the existing architecture (no restructuring, no function signature changes, no merging of utilities). Common opportunities include: early column pruning with `select()`, avoiding repeated `bind_rows()` of the same tables, eliminating unnecessary intermediate assignments, and ensuring vroom lazy loading isn't defeated by premature materialization.

**Primary recommendation:** Review value audit CSVs conversationally with the user to flag inconsistencies worth investigating, then systematically review each script (01-17) for optimization opportunities using profiling-informed prioritization (focus on data loading, large table operations, and repeated logic first).

### Code Optimization Patterns

**Pattern 1: Early Column Selection**
```r
# AVOID: Load all columns, then select subset later
df <- vroom("file.csv", col_types = SPEC)
df_subset <- df %>% select(ID, DX, DX_DATE)  # Materializes ALL columns first

# PREFER: Select columns immediately (if vroom Altrep in use)
df_subset <- vroom("file.csv", col_types = SPEC) %>%
  select(ID, DX, DX_DATE)  # Lazy loading only materializes these 3

# EVEN BETTER: If only need subset, specify in col_types
SUBSET_SPEC <- cols_only(
  ID = col_character(),
  DX = col_character(),
  DX_DATE = col_character()
)
df_subset <- vroom("file.csv", col_types = SUBSET_SPEC)
```

**Pattern 2: Avoid Repeated bind_rows() of Same Tables**
```r
# AVOID: Binding TR1/TR2/TR3 multiple times across multiple functions
has_hodgkin_diagnosis <- function() {
  tr_all <- bind_rows(pcornet$TR1, pcornet$TR2, pcornet$TR3)  # Bind 1
  # ...
}
compute_treatment_dates <- function() {
  tr_all <- bind_rows(pcornet$TR1, pcornet$TR2, pcornet$TR3)  # Bind 2
  # ...
}

# PREFER: Bind once at load time or create helper
pcornet$TUMOR_REGISTRY_ALL <- bind_rows(
  pcornet$TUMOR_REGISTRY1,
  pcornet$TUMOR_REGISTRY2,
  pcornet$TUMOR_REGISTRY3
)
# Then reference pcornet$TUMOR_REGISTRY_ALL in all functions
```

**Pattern 3: Eliminate Redundant Intermediate Assignments**
```r
# AVOID: Assigning to variable only to immediately pipe
result <- df %>% filter(condition)
result <- result %>% mutate(new_col = value)
result <- result %>% select(ID, new_col)

# PREFER: Single chain (readability AND performance)
result <- df %>%
  filter(condition) %>%
  mutate(new_col = value) %>%
  select(ID, new_col)
```

**Pattern 4: Reduce Unnecessary Data Copies**
```r
# AVOID: Multiple copies of large encounter table
enc_copy1 <- encounters %>% filter(condition1)
enc_copy2 <- encounters %>% filter(condition2)
enc_copy3 <- encounters %>% filter(condition3)

# PREFER: Filter once, reuse
enc_filtered <- encounters %>% filter(shared_condition)
enc_subset1 <- enc_filtered %>% filter(specific_condition1)
enc_subset2 <- enc_filtered %>% filter(specific_condition2)
```

**Pattern 5: Use .keep Argument in mutate() to Reduce Columns**
```r
# AVOID: Keeping all columns when only new ones needed
df <- df %>% mutate(new_col = calculation)  # Keeps all columns

# PREFER: .keep="used" to see only inputs/outputs
df %>% mutate(new_col = calculation, .keep = "used")  # Debug view

# PREFER: .keep="none" to drop all original columns
df %>% mutate(new_col = calculation, .keep = "none")  # Only new_col
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Profiling R code | Manual timing with `Sys.time()` | `profvis::profvis()` or `bench::mark()` | Visual flamegraphs, memory profiling, accurate microsecond timing |
| Data validation rules | Custom validation loops | `pointblank` (if needed later) | Rule-based validation engine with reporting; out of scope for Phase 14 but good for v2 |
| Memory usage tracking | Repeated `object.size()` calls | `lobstr::obj_size()` + `pryr::mem_used()` | Accounts for shared references, tracks session memory |
| Finding performance bottlenecks | Guessing based on code inspection | Profile first with `profvis` | 80/20 rule — most time in <20% of code; profile reveals truth |

**Key insight:** Optimization without profiling is premature optimization. For Phase 14, focus on obvious inefficiencies (repeated bind_rows, unnecessary column loads) rather than micro-optimizations that may not matter.

## Common Pitfalls

### Pitfall 1: Assuming vroom is Always Lazy
**What goes wrong:** Certain operations force immediate materialization of vroom Altrep columns, defeating lazy loading and causing memory spikes.
**Why it happens:** Operations like `nrow()`, `summary()`, or non-dplyr base R operations require actual data.
**How to avoid:** Keep operations within tidyverse verbs (`filter()`, `select()`, `mutate()`); check if column materialization happens with `lobstr::obj_addr()`.
**Warning signs:** Unexpectedly high memory usage during "fast" vroom loads; pauses after initial read.

### Pitfall 2: Over-Optimizing Character Columns
**What goes wrong:** Trying to optimize string operations that vroom handles efficiently via Altrep pointers.
**Why it happens:** Assumption that all character data is expensive; vroom only materializes when accessed.
**How to avoid:** Profile first; character columns in vroom are often not the bottleneck (indexes are cheap).
**Warning signs:** Spending time optimizing `str_detect()` when profiling shows I/O or joins are the real issue.

### Pitfall 3: Breaking Named Predicate Readability for Speed
**What goes wrong:** Replacing `has_hodgkin_diagnosis()` with opaque `DT[i, j, by]` data.table syntax to "optimize."
**Why it happens:** data.table is 10-50x faster than dplyr, tempting for large datasets.
**How to avoid:** Per D-06/D-07, preserve named predicate style; only optimize within dplyr/vroom ecosystem.
**Warning signs:** User explicitly requires "reads like clinical protocol" — speed gains aren't worth loss of readability.

### Pitfall 4: Optimizing Scripts Before Profiling
**What goes wrong:** Spending time optimizing scripts that take 0.5 seconds while ignoring a 5-minute data load.
**Why it happens:** Code is easier to "see" than I/O bottlenecks; temptation to start with what's visible.
**How to avoid:** Profile first (even informally with `system.time()`); identify actual slowdowns before optimizing.
**Warning signs:** Making many small changes without measuring impact; no before/after timing comparisons.

### Pitfall 5: Inconsistency as Signal vs. Noise
**What goes wrong:** Flagging every minor variation in value audit as a "data quality issue" when PCORnet CDM allows expected variation.
**Why it happens:** Overzealous interpretation of coding inconsistencies without domain knowledge.
**How to avoid:** Understand PCORnet CDM semantics; e.g., DX_TYPE '09'/'10' distinguish ICD-9 vs ICD-10 (expected), but SEX 'M'/'Male'/'MALE' all meaning male is inconsistency.
**Warning signs:** User saying "that's normal for PCORnet" repeatedly; need to learn which variations are semantic vs syntactic.

### Pitfall 6: Coding Inconsistencies in Derived Variables
**What goes wrong:** Flagging inconsistencies in columns created by the pipeline itself (e.g., `_VALID` flags, harmonized payer categories) as data capture problems.
**Why it happens:** Value audit includes both raw PCORnet columns and derived columns; derived columns follow pipeline logic, not source data variation.
**How to avoid:** Distinguish raw PCORnet columns (potential source inconsistencies) from derived columns (reflect pipeline logic). Per 17_value_audit.R, derived variable audit is separate (payer_summary_derived, hl_cohort_derived CSVs).
**Warning signs:** Questioning why `PAYER_CATEGORY_PRIMARY` only has 9 values when raw `PAYER_TYPE_PRIMARY` has 50+ — that's the harmonization working correctly.

### Optimization Example: Reduce Repeated bind_rows()

**Current pattern in 03_cohort_predicates.R:**
```r
# Source: R/03_cohort_predicates.R lines 68-91
tr_all <- bind_rows(
  if (!is.null(pcornet$TUMOR_REGISTRY1) && "HISTOLOGICAL_TYPE" %in% names(pcornet$TUMOR_REGISTRY1))
    pcornet$TUMOR_REGISTRY1 %>% filter(is_hl_histology(HISTOLOGICAL_TYPE)) %>% distinct(ID)
  else tibble(ID = character()),
  # ... repeated for TR2, TR3
) %>% distinct(ID)
```

**This pattern appears in multiple functions. Optimization:**
```r
# In 01_load_pcornet.R, after loading all tables, create combined TR:
pcornet$TUMOR_REGISTRY_ALL <- bind_rows(
  pcornet$TUMOR_REGISTRY1,
  pcornet$TUMOR_REGISTRY2,
  pcornet$TUMOR_REGISTRY3
) %>% distinct()

# Then in 03_cohort_predicates.R:
tr_hl_patients <- pcornet$TUMOR_REGISTRY_ALL %>%
  filter(
    (exists("HISTOLOGICAL_TYPE") && is_hl_histology(HISTOLOGICAL_TYPE)) |
    (exists("MORPH") && is_hl_histology(MORPH))
  ) %>%
  distinct(ID)
```

**Impact:** Single bind operation instead of 3+ across multiple scripts; ~5-10% speedup in cohort building.

### Optimization Example: Early Column Selection

**Current pattern in 10_treatment_payer.R:**
```r
# Source: R/10_treatment_payer.R compute_payer_at_chemo()
px_dates <- pcornet$PROCEDURES %>%
  filter(PX_TYPE %in% c("CH", "09", "10")) %>%
  filter(str_detect(PX, paste(TREATMENT_CODES$chemo_cpt, collapse = "|"))) %>%
  select(ID, PX_DATE)
```

**Optimized (select early to reduce filter operations on full table):**
```r
px_dates <- pcornet$PROCEDURES %>%
  select(ID, PX, PX_TYPE, PX_DATE) %>%  # Select BEFORE filter
  filter(PX_TYPE %in% c("CH", "09", "10")) %>%
  filter(str_detect(PX, paste(TREATMENT_CODES$chemo_cpt, collapse = "|")))
```

**Impact:** Marginal (vroom Altrep likely already optimizes); only worth if profiling shows PROCEDURES operations are slow.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual frequency tables with `table()` | `count()` + `mutate()` for percentages | dplyr 0.8+ (2019) | Cleaner syntax, tibble output |
| `read.csv()` base R | `vroom::vroom()` with Altrep | vroom 1.0+ (2019) | 10-100x faster, lazy loading |
| Manual validation loops | Rule-based packages (pointblank, dataquieR) | 2020-2023 | Systematic validation, but overkill for exploratory audits |
| Sequential pipe chains | `.by` argument for per-operation grouping | dplyr 1.1.0 (2023) | No `group_by()` / `ungroup()` bookkeeping |
| `if_else()` slow for large data | Optimized `if_else()` / `case_when()` | dplyr 1.2.0 (Feb 2026) | 30x faster, 10x less memory |

**Deprecated/outdated:**
- `tibble::rownames_to_column()`: Use `.rownames` argument in tibble construction (dplyr 1.0+)
- `gather()` / `spread()`: Replaced by `pivot_longer()` / `pivot_wider()` (tidyr 1.0+, 2019)
- `summarise_at()` / `mutate_at()`: Replaced by `across()` (dplyr 1.0+, 2020)

## Open Questions

1. **How much memory do TR bind_rows() operations actually consume?**
   - What we know: TR1/TR2/TR3 are relatively small compared to ENCOUNTER/DIAGNOSIS
   - What's unclear: Whether repeated binding is a real performance issue or negligible
   - Recommendation: Profile with `profvis()` — if TR operations <1% of total time, don't optimize

2. **Are there dead code sections in the 17 scripts?**
   - What we know: Scripts 01-17 were built incrementally across Phases 1-13
   - What's unclear: Whether earlier versions left unused variables, commented-out sections, or redundant checks
   - Recommendation: Systematic review per script, ask user "Is X still needed?" before removing

3. **Which value audit inconsistencies are site-specific vs. global?**
   - What we know: PCORnet has 4 sites (AMS, UMI, FLM, VRT via SOURCE column)
   - What's unclear: Whether coding inconsistencies concentrate in specific sites or spread across all
   - Recommendation: Cross-reference value frequency with SOURCE column to identify site patterns

4. **Should vroom lazy loading be preserved or forced materialization acceptable?**
   - What we know: Vroom Altrep is most efficient when data stays lazy until needed
   - What's unclear: Whether HiPerGator RStudio workflow benefits from lazy loading or if full materialization upfront is simpler
   - Recommendation: Test both approaches; if scripts force materialization anyway, explicit eager loading may be clearer
