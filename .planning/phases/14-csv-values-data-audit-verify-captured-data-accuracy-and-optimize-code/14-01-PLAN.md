<task type="checkpoint:decision" gate="blocking">
  <name>Task 2: Conversational review of value audit CSVs for coding inconsistencies</name>
  <read_first>
    - R/17_value_audit.R (understand CSV format and column structure)
    - R/00_config.R (understand expected code values: ICD codes, payer mappings, treatment codes)
    - R/01_load_pcornet.R (understand column type specs and _VALID flag patterns)
  </read_first>
  <action>
    Claude reads the user-provided CSV files and analyzes them for coding inconsistencies per D-03.

    Focus areas (per D-03 and RESEARCH.md):
    1. SAME CONCEPT, DIFFERENT REPRESENTATIONS:
       - Sex coded as 'M'/'Male'/'MALE' or 'F'/'Female'/'FEMALE'
       - Race values with inconsistent coding (numeric codes vs text labels)
       - Yes/No as 'Y'/'YES'/1/'N'/'NO'/0
       - Hispanic as 'Y'/'N'/'R'/'NI' vs full text

    2. MIXED DATE FORMATS in same column:
       - YYYYMMDD vs YYYY-MM-DD in raw character values
       - SAS epoch dates (1899-12-30) appearing as valid dates

    3. ICD CODE FORMAT INCONSISTENCIES:
       - Dotted (C81.10) vs undotted (C8110) in same DX column
       - Leading zeros present/absent (09 vs 9 for DX_TYPE)

    4. PAYER CODE VARIATIONS:
       - Multiple codes mapping to same payer category
       - Unexpected sentinel values beyond NI/UN/OT

    5. SENTINEL VALUE INCONSISTENCIES:
       - 'NI', 'UN', 'OT', 'UNKNOWN', NA used interchangeably
       - Empty strings vs NA vs sentinel codes

    What to SKIP (expected PCORnet CDM variation, not inconsistencies):
    - DX_TYPE '09' vs '10' (ICD-9 vs ICD-10 era split -- expected)
    - PX_TYPE 'CH', '09', '10', 'RE' (different code systems -- expected)
    - ENC_TYPE variation (AV, ED, IP, etc. -- PCORnet standard)
    - Multiple SOURCE values (AMS, UMI, FLM, VRT -- site identifiers)
    - Derived _VALID columns showing TRUE/FALSE (pipeline-generated)
    - Harmonized payer categories in derived audits (pipeline-generated)

    Discussion format:
    - Present findings grouped by table
    - For each finding: quote the values, counts, and percentages
    - State whether it appears to be a data capture problem or expected variation
    - Ask user if they want to act on each finding

    Per D-09: No separate findings report or markdown output -- findings discussed live in conversation.
  </action>
  <acceptance_criteria>

    - User has stated which findings (if any) they want to act on
  </acceptance_criteria>
  <decision>Which coding inconsistencies (if any) should be fixed via code changes in the pipeline?</decision>
  <options>
    <option id="no-changes">

      <cons>Potential real inconsistencies left unaddressed</cons>
    </option>
    <option id="normalization-changes">
      <name>Add normalization steps for specific columns</name>
      <pros>Cleaner downstream analysis; consistent values</pros>
      <cons>May mask original data variation that has clinical meaning</cons>
    </option>
    <option id="documentation-only">
      <name>Document findings but don't change code</name>
      <pros>Preserves raw data fidelity; creates audit trail</pros>
      <cons>Inconsistencies remain in analysis outputs</cons>
    </option>
  </options>
  <resume-signal>Select: no-changes, normalization-changes, documentation-only, or describe a custom approach</resume-signal>
