## Approach

Apply the Plan 31-01 pattern to each script in order of expected impact. Start with the heaviest (Phase 22's all-site duplicate detection and Phase 25's multi-source overlap — both do self-joins across 5 sites), since those are where the DuckDB speedup matters most and where translation gaps are most likely.

Do one script fully — migrate, parity test, benchmark, commit — before starting the next. This avoids a half-migrated pile if a translation gap takes unexpected time.

Parity test via `tools::md5sum()` on sorted CSV outputs. Load both CSVs, sort by all columns, write to normalized paths, md5. Any diff triggers `waldo::compare()` investigation.

## Steps (Per Script)

1. Read the script, identify every `load_pcornet_table()` call and every predicate invocation
2. Open DuckDB connection at script start, `on.exit()` close
3. Replace `load_pcornet_table()` with `get_pcornet_table(name, con)`
4. Leave predicate logic unchanged unless a translation gap surfaces
5. Move `materialize()` as late as possible — ideally right before `save_output_data()` or `write_csv()`
6. Run under `USE_DUCKDB = TRUE`; capture CSV outputs to parity-test paths (not overwriting originals)
7. Sort both RDS and DuckDB CSVs by all columns; `md5sum` comparison
8. If md5s match: parity achieved, proceed to benchmark
9. If md5s differ: `waldo::compare()` on loaded data frames, diagnose, fix or document tolerance
10. Benchmark: 3 runs each backend via `benchmark_script()`, append to `output/logs/duckdb_benchmark.csv`
11. Commit migrated script + parity evidence + benchmark rows
12. Move to next script

## Script-Specific Notes

- **`R/22_multi_source_overlap_detection.R`** (Phase 25): Self-join with 7-day window predicate. `dbplyr` should translate this as a self-join with `ABS(DATE1 - DATE2) <= 7`. Watch for DuckDB's date arithmetic returning INTERVAL rather than integer days — may need explicit cast.
- **`R/24_per_patient_source_detection.R`** (Phase 28): Uses `data.table` per the phase plan for speed. Decision point: migrate to DuckDB anyway for consistency, or retain `data.table` as a documented exception. Benchmark both against each other before deciding. If `data.table` wins on this specific workload by a notable margin, retain it and document in the migration guide.
- **`R/21_all_site_duplicate_dates.R`** (Phase 22): Cross-site pairwise comparison. Similar self-join shape to Phase 25, similar translation concerns.
- **`R/20_all_source_missingness.R`** (Phase 21): Cross-tab heavy, no self-joins, should translate cleanly. Expected easy migration.
- **`R/23_overlap_classification.R`** (Phase 26): Depends on Phase 25 output. Classification via `case_when()` should translate fine.

## Parity Test: HIPAA Suppression Edge Case

If unsuppressed counts match but suppressed output differs (e.g., a cell reads "<11" under one path and "9" under the other), that's a sort-stability artifact, not a real diff. Normalize: sort underlying data before suppression is applied, then compare. If unsuppressed counts differ at all, that's a hard fail requiring investigation.

## Notes

- If any script takes significantly longer than 2 hours to migrate, stop and assess — likely a translation gap needs structural rework rather than a quick fix. Escalate by filing a blocker and moving to the next script.
- Resist the urge to refactor while migrating. If a script has cleanup opportunities visible during migration, file them for a separate cleanup phase. Migration scope is "change the data source, nothing else."
