# Plan 32-02: Speedup Report, Migration Guide, Flip Default

**Phase:** 32 (Diagnostic Scripts DuckDB Migration & Benchmarks)
**Requirements:** DBDIAG-03, DBDIAG-04
**Estimated effort:** 3–4 hours

## Inputs

- `output/logs/duckdb_benchmark.csv` populated by Plans 31-02 and 32-01
- All migrated scripts from Plan 32-01 with parity passing
- `docs/DUCKDB_TRANSLATION_NOTES.md` with all accumulated gap documentation

## Outputs

- `R/26_generate_speedup_report.R` — script that reads benchmark CSV and produces the report
- `output/reports/duckdb_speedup_report.md` — generated report
- `docs/DUCKDB_MIGRATION_GUIDE.md` — new, standalone guide for future script authors
- `R/00_config.R` updated: `USE_DUCKDB <- TRUE` as default with deprecation comment
- Final HiPerGator end-to-end verification: full pipeline run with `USE_DUCKDB = TRUE`

## Approach

Report generation is deterministic: read the benchmark CSV, compute medians, compute speedup ratios, format as markdown. Script rather than manual composition so the report can be regenerated on future extracts without rewriting prose.

Migration guide is the one piece of prose worth writing by hand. Audience is future-you (or a teammate) adding a new diagnostic script six months from now. Keep it short — connection pattern, dispatcher, translation gotchas, template, parity test methodology.

Flipping the default is a one-line diff. The careful part is the comment around it: explicit deprecation timeline, pointer to the migration guide, and a clear statement that the RDS path still works for anyone who needs to bisect against historical behavior.

## Steps

1. Write `R/26_generate_speedup_report.R`:
   - Read `output/logs/duckdb_benchmark.csv`
   - Group by (script, backend), compute median duration and median peak memory
   - Compute per-script speedup (RDS median / DuckDB median) and memory reduction
   - Format as markdown table with supplementary prose
   - Write to `output/reports/duckdb_speedup_report.md`
2. Run the report generator; review the output; add a milestone summary paragraph at the top
3. Write `docs/DUCKDB_MIGRATION_GUIDE.md` with sections:
   - **Overview** — what changed and why
   - **Connection pattern** — `open_pcornet_con()` + `on.exit()` idiom
   - **`get_pcornet_table()` and `materialize()`** — how to use them
   - **When to call `collect()` / `materialize()`** — as late as possible
   - **Known translation gaps** — pointer to `DUCKDB_TRANSLATION_NOTES.md` plus top 2–3 to watch for
   - **Template script** — 30-line skeleton showing the full pattern
   - **Parity test methodology** — how to verify a new script against an RDS baseline
4. Update `R/00_config.R`: change `USE_DUCKDB <- FALSE` to `USE_DUCKDB <- TRUE`; add a 3–4 line comment noting the deprecation timeline and the pointer to the migration guide
5. Run the full pipeline end-to-end on HiPerGator with the new default — cohort build + all 5 diagnostic scripts — and confirm everything still works (spot-check outputs against expected shapes)

## Success Criterion Check

From Phase 32 CONTEXT: speedup target is ≥ 3× on at least 3 of 5 scripts. Check this explicitly in the report; call it out in the milestone summary. If met, the milestone closes with a green flag. If not, document which scripts fell short and why — likely candidates are scripts dominated by CSV writing or HIPAA suppression overhead rather than query cost. Not meeting the target is not a blocker; it's honest signal about where the speedup lives.

- `docs/DUCKDB_MIGRATION_GUIDE.md` exists, has all 6 sections, template compiles and runs

## Notes

- The speedup report should be honest about any script that regressed or didn't meet the 3× target. Hiding bad numbers in a report is worse than showing them.
- The migration guide's template is the highest-leverage piece — it's what a future script author will copy-paste. Make it actually copy-pasteable (real code, not pseudocode), commented, tested.
- Flipping the default should be the last thing done, after everything else in this plan is green. If any issue surfaces during the final pipeline run, revert to `FALSE` and investigate.
