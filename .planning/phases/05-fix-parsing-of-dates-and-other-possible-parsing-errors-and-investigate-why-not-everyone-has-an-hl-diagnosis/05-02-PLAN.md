**SECTION 1: Date Parsing Failures Audit (D-01, D-02)**

For each of the 9 loaded tables (per D-01: ALL 9):
1. Get column names matching the date detection regex from 01_load_pcornet.R
2. For each date column, check if it's actually parsed as Date type
3. If still character: extract unique unparsed values with counts
4. If Date type but has NAs: count NAs (these could be legitimate nulls OR parse failures)
5. For Date columns, compare NA count vs total to compute parse failure rate

Write results to `output/diagnostics/date_parsing_failures.csv` with columns: table, column, type_detected, n_total, n_na, na_percent, sample_raw_values (for character columns only).

Also compare parsed date ranges against sanity bounds: dates before 1900-01-01 or after current date are suspect. Write date range issues to `output/diagnostics/date_range_issues.csv` with columns: table, column, n_before_1900, n_future, min_date, max_date.

Console output via message(): per-table summary showing number of date columns found, number with issues, and overall parse rate.

**SECTION 2: Column Detection Regex Audit (D-03)**

Read csv_columns.txt and extract all column names from ALL 22 tables (not just the 9 loaded). For each column name, check whether it matches the date detection regex `(?i)(DATE|^DT_|^BDATE$|^DOD$|^DT_FU$|DXDATE|_DT$|RECUR_DT|COMBINED_LAST_CONTACT|ADDRESS_PERIOD_START|ADDRESS_PERIOD_END)`.

Produce a comparison report:
- Columns that match the regex (expected date columns)
- Columns that DON'T match but MIGHT be dates (heuristic: column contains time-like values or name suggests temporal data)

Write to `output/diagnostics/date_column_regex_audit.csv` with columns: table, column, regex_match (TRUE/FALSE), notes.

Console output: count of matched vs unmatched columns, list of any potentially missed date columns.

**SECTION 3: Column Type and Missing Value Audit (D-15, D-16, D-17, D-19)**

For each of the 9 loaded tables:

(D-15) **Type mismatch check:** For tables with explicit col_types specs (ENROLLMENT through DEMOGRAPHIC), compare the spec column names against actual loaded column names. Flag columns present in spec but missing from data, and columns in data but not in spec. For TUMOR_REGISTRY tables (which use .default = col_character()), flag columns that look numeric (all non-NA values pass `grepl("^-?\\d+\\.?\\d*$", x)`) or date-like but are loaded as character.

(D-16) **Missing/extra columns:** Compare expected columns (from col_types specs in 01_load_pcornet.R) against actual columns in each table. Write discrepancies to `output/diagnostics/column_discrepancies.csv` with columns: table, column, status (missing_from_data, extra_in_data, type_mismatch), expected_type, actual_type.

(D-17) **Encoding check:** For character columns, check for non-ASCII characters using `grepl("[^\\x00-\\x7F]", x)`. Check first row for BOM marker (`grepl("^\\xEF\\xBB\\xBF", x)`). Report counts of affected columns per table.

(D-19) **TUMOR_REGISTRY type audit:** For TR1 (314 cols), TR2 (140 cols), TR3 (140 cols), sample first 100 non-NA values of each character column. Flag columns where >80% of sampled values look numeric or date-like. Write to `output/diagnostics/tr_type_audit.csv` with columns: table, column, current_type, n_sampled, pct_numeric, pct_datelike, recommendation.

Write combined missing value audit to `output/diagnostics/missing_values_audit.csv` with columns: table, column, type, n_total, n_missing, pct_missing. Only include columns with >10% missing.

Console output per section: number of columns with issues, summary of most problematic columns.
