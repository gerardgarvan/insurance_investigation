**SECTION 3: DuckDB COUNT QUERIES**
For each entry in code_type_map, query the appropriate PCORnet table(s) to get per-code record count and patient count.

Strategy: Group by source_table type and batch queries.

For PROCEDURES (PX_TYPE = "CH", "09", "10", "RE"):
- Load table once: proc_tbl <- get_pcornet_table("PROCEDURES")
- For exact match vectors: filter(PX_TYPE == px_type, PX %in% codes)
- For prefix match vectors: use a SQL-compatible approach. Since DuckDB supports LIKE, for each prefix code, match PX values starting with that prefix. Approach:
  - Collect matching PX values first by iterating prefixes with str_detect-like logic
  - OR use a subquery approach: pull all PX values with the right PX_TYPE, then locally match prefixes
  - Recommended: For prefix vectors (chemo_icd10pcs_prefixes, radiation_icd10pcs_prefixes, cart_icd10pcs_prefixes), the codes ARE the full codes stored in config (like "3E03305", "3E04305", etc.) -- they are called "prefixes" because they represent ICD-10-PCS root codes that may have longer variants in data. Check actual config values. If they are 7-character codes, they may actually be exact matches against PX column. Inspect the actual values in config to determine. If the values look like "3E0" (3 chars) then prefix match; if "3E03305" (7 chars) then likely exact match. Use the appropriate matching strategy.
- group_by(code = PX) %>% summarise(records = n(), patients = n_distinct(ID))
- collect()

For PRESCRIBING + MED_ADMIN (RXNORM vectors):
- CRITICAL: To avoid double-counting patients who appear in both tables, use the combined approach from RESEARCH.md Pitfall 2:

- enc_tbl <- get_pcornet_table("ENCOUNTER")
